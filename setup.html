You'll install a terminal tool, log in, and connect three data sources. Copy-paste commands — that's it.

Why bother with a CLI at all? Don't I already have M365 Copilot?

Quick analogy: M365 Copilot is the librarian — you ask, it finds, it reads back. CLI Copilot is the contractor — you give it a job, it goes off and does it.

M365 Copilot is great at what it sees: your email, calendar, Office files. Ask it about a thread, it summarizes. Ask it to draft a reply, it does. But it lives inside the M365 surface and stays there.

CLI Copilot is a different shape. It runs on your machine, in your terminal, and it actually does things. Runs commands. Edits files. Calls APIs. Ties tools together.

What the CLI gives you that M365 Copilot doesn't:

  1. Reach beyond M365. Through MCP connectors, you plug in anything that exposes one: PMX, GitHub, your own internal services, partner APIs, local repos. M365 Copilot is fixed to what Microsoft ships. The CLI is open.
  2. Execute, don't just retrieve. "Pull last week's PMX projects, draft a one-pager for each, save to OneDrive" — ask once, it runs the whole chain. M365 Copilot can find and summarize. The CLI can do the work.
  3. Speak every other command-line tool for you. Azure CLI, GitHub CLI, git, npm. You don't need to memorize syntax. Tell Copilot what you want; it picks the tool and runs it.
  4. Live where your real work happens. Your repos, your scripts, your files, your terminal. No tab-switching, no re-uploading.

Set up once. Use it for real work after.

A taste of what's possible: "Find my last three meetings with Contoso, summarize what we discussed, check their PMX project status, then draft a follow-up email for me to review before sending." Calendar + PMX + Outlook + Copilot reasoning, chained in one prompt. That's the CLI.

Already comfortable with CLI tools? Skim the steps and jump to Step 5: Verify everything. If all five checks pass, you're done.

Never opened a terminal before? Perfect — that's exactly who this is for. Every step explains what's happening and why. Just follow along and copy-paste the commands.

Step 0

Get your GitHub account ready

Copilot CLI runs on GitHub. To use it, you need a GitHub account with Copilot access. As a Microsoft employee, this is provided for you — but you may need to set it up.

First: find out what you have

Go to aka.ms/copilot and sign in with your Microsoft credentials. This portal shows your Copilot entitlement and your Enterprise Managed User (EMU) — a corporate GitHub account Microsoft created for you. It usually looks like youralias_microsoft.

Take note of the EMU username shown there. You'll need to know this exists, even if we won't use it as the primary account for this session.

Do you have a personal GitHub account?

If you're not sure, go to github.com/login and try signing in. If you can't, you need to create one.

📝 No GitHub account yet? Create a free personal account at github.com/signup. Use any email you like — this is your personal account, not a Microsoft-managed one.

You might end up with two GitHub accounts

This is normal at Microsoft. You may have seen your EMU at aka.ms/copilot above. Here's the difference:

AccountWhat it isHow to tell
PersonalYour own GitHub account — you created it yourselfUsername is whatever you chose (e.g., johansen42)
EMUA corporate account Microsoft created for youUsername ends in _microsoft (e.g., johansen_microsoft)

You might have both, just one, or neither yet. That's fine.

What's an EMU and why do we need two GitHub accounts?

EMU = Enterprise Managed User. A corporate GitHub account Microsoft created and manages for you. The username always ends in _microsoft.

Why does this exist? Microsoft has a lot of internal code, tools, and partner data sitting in private GitHub repos. EMU accounts are the security boundary — only _microsoft users can see those repos. Personal accounts can't.

Your personal account is yours. Your email, your career identity, your open-source work, your side projects. Microsoft doesn't manage it.

For the workshop:

  • Personal account is your default. Almost everything we do today uses it.
  • EMU account only comes out when you install something from a Microsoft-internal repo (PMX is the example).

The pattern is: switch to EMU → install → switch back to personal. Both stay logged in; you just toggle which one is active.

Which account for this session?

We recommend your personal account. It's simpler to use and works with everything we'll do in the session.

Connect your account to the Microsoft Copilot entitlement

You already visited aka.ms/copilot above. If the portal showed a green checkmark or your EMU, the entitlement is active. Now connect your personal account to the same license:

👉 Action: Go to copilot.github.microsoft.com. Sign in and follow the on-screen steps to link your personal GitHub account to the Microsoft Copilot license. If you see a confirmation — you're good.
⚠️ Can't access that portal? Search "GitHub Copilot QuickStart" on the R&I SharePoint site — it has step-by-step instructions and an automated verification tool. Or ask your manager or a colleague who already uses Copilot CLI.
Step 1

Install command-line tools

First: open PowerShell

PowerShell is a text-based tool for talking to your computer — like File Explorer, but you type commands instead of clicking. It comes pre-installed on Windows.

Press the Windows key, type PowerShell, and open the one that says "PowerShell 7" or "pwsh". Avoid the one called "Windows PowerShell" — it's an older version.

⚠️ Don't see PowerShell 7? You might only have the old version. Open "Windows PowerShell" instead — it will work for the install commands. We'll sort out the version later.
What's the difference between PowerShell 7 and Windows PowerShell?

Both are command-line tools that come with Windows. "Windows PowerShell" is the older version (5.x) — shipped with the OS for years. "PowerShell 7" (sometimes called "pwsh") is the newer cross-platform version Microsoft ships separately.

They look almost identical but PowerShell 7 has better tab completion, better error messages, and supports modern command patterns. For this workshop either one works — but if you have PowerShell 7, use it.

Then: install five small tools

Copilot CLI doesn't work alone. It calls four other tools behind the scenes — Node.js (its runtime), Git (for source control), GitHub CLI (for account switching), and Azure CLI (for PMX login). None of them have buttons or windows of their own; they sit underneath, ready to be called from your command line.

Run these one at a time, waiting for each to finish before pasting the next. (Pasting them all in one go can break — PowerShell handles single-line pastes most reliably.) Each takes 30–60 seconds.

winget install OpenJS.NodeJS.LTS
winget install Git.Git
winget install GitHub.cli
winget install Microsoft.AzureCLI
winget install GitHub.Copilot
💡 Pop-ups are normal. Windows will ask "Do you want to allow this app to make changes?" — click Yes. Git's installer also opens a wizard with several screens — click Next through all defaults. This is expected.
What is winget, anyway?

It's Windows' built-in app installer — like the Microsoft Store, but you control it from PowerShell. Type winget install <tool> and it downloads and installs the official version straight from the publisher (Microsoft, GitHub, Git, etc.).

Microsoft ships it pre-installed on Windows 11. Think of it as the same idea as apt-get on Ubuntu or brew on Mac, if those names mean anything to you.

When all five are done, close and reopen PowerShell — this is important. The new commands won't be available until you do.

Verify they all worked

In the fresh PowerShell window, run these one at a time:

node --version
git --version
gh --version
az --version
copilot --version
✅ Pass: Each command prints a version number. The exact numbers don't matter — what matters is that none of them say "not recognized".
⚠️ winget not found? Open the Microsoft Store app, search for "App Installer", and install it. Then restart PowerShell and try again.
⚠️ Something says "not recognized" after the verify step? Make sure you closed and reopened PowerShell. If still missing, restart your computer and try the verify commands again.
💡 Can't install at all? Corporate policies blocking the install? You can use GitHub Codespaces as a backup — it runs in your browser with most tools pre-installed. Tell the facilitator if you need this option.
Step 2

Log in to GitHub and Copilot

Two separate tools both need a GitHub login: GitHub CLI (so you can switch accounts later) and Copilot CLI (so the AI knows who you are). We'll do them in order.

2a. Log into GitHub CLI

Start the login flow:

gh auth login

It asks four questions, one at a time. Use the arrow keys to highlight the right answer, then press Enter:

  • What account do you want to log into? → GitHub.com
  • What is your preferred protocol for Git operations? → HTTPS
  • Authenticate Git with your GitHub credentials? → Yes
  • How would you like to authenticate GitHub CLI? → Login with a web browser

It shows a one-time code and offers to open a browser. Press Enter — your browser opens. If GitHub asks you to sign in first, do so with your personal GitHub account (this happens when the browser isn't already signed in — it's normal). Then paste the code and authorize.

When the page says "Congratulations, you're all set," come back to PowerShell.

⚠️ Don't re-run gh auth login if you hit a snag. Every new run invalidates all previous codes. If the browser is stuck, looks wrong, or shows "Uh oh, we couldn't find anything", just go to github.com/login/device and paste the code from your terminal — the existing one is still valid. Only the most recent code in your terminal works; older ones are dead.
💡 Got redirected to github.com/login instead of /login/device? That's GitHub's pre-auth redirect — sign in on that page and GitHub will send you to the device-code page automatically.
✅ Pass: PowerShell prints ✓ Logged in as <your-username>. Confirm anytime with gh auth status.
Why HTTPS and not SSH?

Both are ways for your computer to talk securely to GitHub when pushing or pulling code. HTTPS uses your username and the token you just authenticated with — simple, works through corporate proxies, no setup needed.

SSH uses cryptographic keys you'd have to generate and add to your GitHub account first — more powerful, more steps. For this workshop, HTTPS is the easy answer. You can switch later if you ever want to.

Why am I logging in twice — gh AND copilot?

They're two separate tools that both need to know who you are. GitHub CLI (gh) is for talking to GitHub from PowerShell — switching accounts, cloning repos, opening pull requests. Copilot CLI (copilot) is the AI assistant.

Same GitHub account, but each tool keeps its own login session. Logging into gh first also means Copilot CLI can use it later for things like installing the PMX plugin in Step 4.

2b. Log into Copilot CLI

Now launch Copilot CLI:

copilot

You'll see an animated banner. Inside Copilot CLI, type:

/login

A browser window opens — sign in with your personal GitHub account again. Once logged in, test with a simple prompt:

What day is it today?
✅ Pass: You get an answer back. Type /exit to leave for now.
⚠️ Login fails or times out? Make sure you're on the corporate network or VPN. Verify your Copilot subscription at github.com/settings/copilot.
Step 3

Log in to Azure (for PMX)

The PMX MCP server needs an Azure login to talk to Dynamics 365. Azure CLI was already installed in Step 1 — this is just the sign-in.

Log in to the Microsoft tenant

az login --tenant 72f988bf-86f1-41af-91ab-2d7cd011db47

A browser window opens. Sign in with your Microsoft account. Done when it says "You have logged in."

📝 Subscription picker: After signing in, the terminal lists your Azure subscriptions and asks you to pick one. Just press Enter to accept the default — PMX doesn't care which subscription you pick.

✅ Pass: Run this — you should see your account name in the output:

az account show
Step 4

Connect your tools

MCP servers and plugins are connectors that let Copilot CLI talk to external systems — like Power Automate connectors, but for CLI. We'll connect three:

  1. GitHub — repos, issues, pull requests (built into Copilot CLI)
  2. M365 — email, calendar, Teams (install in 4a)
  3. PMX — your partner management data in Dynamics 365 (install in 4b)

Copilot CLI ships with GitHub's MCP server already built in — no setup needed. Step 4a installs M365 as plugins for your personal account. Step 4b installs PMX, which needs your Microsoft EMU account because the repo is internal to Microsoft.

What is MCP, in plain language?

MCP stands for "Model Context Protocol" — but ignore the name. Think of it as adapters that plug Copilot CLI into your other tools. Each adapter lets Copilot read and act on a specific system: PMX for partner data, GitHub for repos, M365 for email/calendar/Teams.

Without adapters, Copilot is a smart chatbot that can only talk. With adapters, it can actually look up your projects, send emails, post in Teams. Same Copilot — just with hands.

4a. Set up M365 (personal account)

First, confirm your personal account is the active GitHub account:

gh auth status

You should see Active account: true next to your personal username. If a different account is active, switch:

gh auth switch --user <your-personal-username>

Then launch Copilot CLI:

copilot

Inside Copilot CLI, paste this prompt:

Set up M365 capabilities (Outlook, Teams, Calendar, SharePoint) in my Copilot CLI. If anything is already installed, just confirm it. Then run a quick test — fetch my next calendar meeting and one of my recent GitHub repos.

Copilot will install whatever's missing, then run the tests. The GitHub test confirms the built-in GitHub MCP server is working too.

⚠️ If Copilot asks "Where do you want to configure these MCP servers?" — pick 2. Copilot CLI. The other options configure MCP for VS Code's Copilot Chat or other tools, not your local CLI.
✅ Pass: Copilot returns your next meeting and a GitHub repo. Both M365 and GitHub are alive.

When done, leave Copilot CLI:

/exit

4b. Install PMX (Microsoft EMU account)

PMX lives in gim-home/pmx-mcp, which only your _microsoft EMU account can see. The flow: log in to gh as your EMU (you already logged in as personal in Step 2a), then let Copilot CLI orchestrate the install for you.

First: log in as your EMU

Run gh auth login again — same four answers as Step 2a (GitHub.com, HTTPS, Yes, Login with a web browser). When the browser opens, sign in as your _microsoft EMU this time (Microsoft sign-in, not your personal GitHub).

gh auth login

Then verify both accounts are now known to gh:

gh auth status

You should see two entries — your personal account and your _microsoft EMU. Both must be present before continuing. (Either one can be active; the prompt below will set it.)

Now: let Copilot install PMX

Launch Copilot CLI:

copilot

Inside Copilot CLI, paste this prompt:

Install the PMX MCP server for me. I have two GitHub accounts known to `gh` — my personal one and my Microsoft EMU (username ends in `_microsoft`).

1. Run `gh auth status` to confirm both accounts are present and note their usernames.
2. Switch the active gh account to the `_microsoft` EMU.
3. Run `copilot plugin marketplace add gim-home/pmx-mcp`.
4. Run `copilot plugin install pmx-mcp@pmx-mcp`.
5. Switch the active gh account back to my personal one.
6. Run `gh auth status` again to confirm my personal account is now active.

If any step fails, stop and tell me what went wrong — don't continue. When everything succeeds, tell me to type `/exit` and run `copilot` again so the new MCP server registers.

Copilot will run each shell command, switching the active gh account as it goes. When it finishes, it will tell you to relaunch.

Exit Copilot CLI:

/exit

And start it again so the new MCP server registers:

copilot
⚠️ Both accounts must be logged in before you paste the prompt. The prompt switches between them — it does not create them. If gh auth status only shows one account, run gh auth login for the missing one first.
💡 No _microsoft EMU account? Skip 4b — only PMX needs it, and we can demo PMX together at the session. Steps 0–4a are enough to participate.
Manual fallback — if the Copilot prompt fails

If Copilot can't run shell commands or the prompt misfires, do it yourself. Run them one at a time.

1. Find your EMU username. Run:

gh auth status

Look at both account lines. Your EMU username is the one ending in _microsoft (e.g. jeghammer_microsoft). Copy that exact name — you'll need it next.

2. Switch the active account to your EMU. Replace YOUR_EMU_USERNAME with the name you copied above:

gh auth switch --user YOUR_EMU_USERNAME

3. Confirm the switch worked. Run gh auth status again and check that the line marked "Active account: true" is on your _microsoft entry. If it's still on your personal account, step 2 didn't take — re-run it.

gh auth status

4. Install PMX. With the EMU active, run these in order:

copilot plugin marketplace add gim-home/pmx-mcp
copilot plugin install pmx-mcp@pmx-mcp

5. Switch back to your personal account. Replace YOUR_PERSONAL_USERNAME with your personal GitHub username:

gh auth switch --user YOUR_PERSONAL_USERNAME
⚠️ Got 403 Write access to repository not granted? Your personal account is still active. The marketplace add command is using personal creds against a Microsoft-internal repo. Re-run step 2 (the gh auth switch), verify with gh auth status that the EMU is active, then re-run the marketplace command.

Same end state as the prompt path. The EMU must be the active account when marketplace add runs — that command fetches the manifest using your active gh credentials. Personal account → 403 or 404, no auto-recovery.

⚠️ Having trouble? MCP setup depends on your environment and permissions. If something fails, don't stress — we'll troubleshoot together at the start of the session. Just make sure Steps 0–3 are done.
