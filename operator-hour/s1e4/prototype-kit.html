Step 1 · the substance

Start from the recap start here

The substance is already written for you. Teams makes an AI recap of every call: the summary, the decisions, the action items, all in the customer's own words. So don't sit and fill in a template from memory. Grab that recap, drop it in your project folder as recap.md, and let the build read it straight. That is what makes the prototype feel like theirs.

Got their data? Drop in their catalog, price list, or product table as a file too, even a rough one. No file? Invent rows that match their shape. Real structure, fictional records, never their actual data.

Only got a long transcript instead of a clean recap? Run this first to boil it down to what a prototype actually needs.

Prompt 0 · distil the call (optional) 
Read transcript.md. Pull out only what I'd need to build a prototype,
and write it to recap.md:

- The problem in one sentence, in their words.
- The three or four hard constraints (environment, policy, deadline, scope).
- Five phrases they actually used, so the prototype speaks their language.
- The one moment that would make them lean in.

Keep it to one page. No filler.
Step 2 · the build

The build prompt the engine

Open Copilot CLI in the folder with your recap and data file, paste this, and adjust the bracketed parts. The constraints at the end are what keep the output simple enough to demo: one file, no installs, opens in any browser.

Prompt A · build the prototype 
Read recap.md and [data file].

Build me a clickable mock of [the thing] as one HTML file, prototype.html:

- A welcome screen in [customer]'s colours.
- The two or three things their users actually come to do, as big choices.
- Content drawn from the data file. Their structure, their names.
- A clearly visible "prototype" label in the footer.

No installs, no frameworks, one file. Open it in the browser when you're done.

Step 3 · the handover

Make it the first commit the multiplier

A prototype trapped on your laptop is a dead end. Ask for the handover in the same conversation, and the customer's engineers can stand it up in their own subscription and keep building. This is what turns a demo into an engagement.

Prompt B · write the handover 
Write the handover for this prototype.

- A README an engineer can follow cold: what it is, what it proves,
  how to run it, and what's mocked.
- The infrastructure as code (Bicep) so they can deploy it to their
  own subscription.
- A short "known shortcuts" list: everything demo-grade that needs
  real engineering before production.

Then verify the README by following it yourself from a clean folder.
The three rules, always: fictional data only (their shape, never their records). Say it's a prototype, on screen and out loud. And it reaches production through engineers, a repo and a review, never straight from your demo laptop.
