Episode 6 · Take-home kit

Build the agent.

Otto is one repo. Fork it, swap the persona, drop in your team's knowledge, deploy. Your team has a sidekick in a Teams chat by Friday. Three ingredients, no engineer, under thirty dollars a month to run.

Ingredient 1 · persona How it talks One markdown file. Who it helps, how it sounds, what it refuses.
Ingredient 2 · knowledge What it knows Your team's stuff. The FAQ and handbook you explain twice a week.
Ingredient 3 · a home Where it lives A Teams chat. It answers when someone @mentions it.
The repo. Everything behind Otto and Remy is one repo: github.com/JW-Sthlm/agent-otto. Clone it, then follow the four steps below. The whole thing is TypeScript plus a few markdown files, deployed with one command. You change the markdown, not the code.
Step 1 · fork it

Clone the repo 2 minutes

Start from the working thing, not a blank folder. Clone Otto, rename it to your agent, and let Copilot walk you through what each part does. You don't need to read the code, you need to know which two folders to edit (the persona and the knowledge).

Prompt 1 · clone and rename 
Clone https://github.com/JW-Sthlm/agent-otto into a new folder called agent-[your-name].

Then walk me through the repo in plain English. I'm not a developer. Tell me:

- Which one file holds the agent's personality.
- Which folder holds what the agent knows.
- The one command that deploys it.

Don't change anything yet. Just orient me.
Rename it so it isn't Otto. Give your agent its own name in src/prompts/system.md and teamsapp/manifest.template.json. Two agents from the same repo only differ by their persona and their knowledge. That's the whole point.
Step 2 · the persona

Write how it talks one file

The persona is a single markdown file, src/prompts/system.md. Plain English. Who it serves, how it sounds, what it does and doesn't touch. Keep Otto's shape and just swap the job. Otto's persona is shorter than most emails you send.

Prompt 2 · write the persona 
Open src/prompts/system.md. Keep its structure, but rewrite it for my agent:

- Name: [agent name].
- Who it helps: [your team, e.g. new joiners at our consultancy].
- What it answers: [the domain, e.g. onboarding, where things live, who to ask].
- Voice: plain, helpful, dry. Zero em dashes. No corporate filler.
- Honesty: when it doesn't know, it says "I don't know that yet" instead of guessing.
- Boundaries: no legal, HR, or compliance advice. No confidential data.

Show me the rewritten file. Don't touch anything else.
Step 3 · the knowledge

Drop in what it knows your stuff

This is the part that makes it yours. The src/knowledge/ folder is just markdown files, loaded into the agent at startup. Delete Otto's course files, drop in your team's stuff: the onboarding FAQ, the handbook, the playbook nobody can find, the questions you answer twice a week. Swap these files and you have a different agent.

Prompt 3 · swap the knowledge 
In src/knowledge/, delete the existing files. I'm going to give you my team's content instead.

I'll paste or point you at: [an onboarding FAQ, a team handbook, a checklist, a partner FAQ, whatever I keep re-explaining].

For each one, write a clean markdown file in src/knowledge/. Keep my facts exact.
Don't invent answers, don't add anything I didn't give you, and flag anything that
looks confidential so I can decide whether it belongs in here.

Then list the files you created.
The agent answers from these files. Wrong file, wrong answer, in front of your team. You own the knowledge, so keep it true and check it the way you'd check anything going out with your name on it.
Step 4 · ship it

Deploy, then pick a home one command

Deploy with one command, then put the agent where your people already are. Three publish paths, same three ingredients. Pick the door that fits you. And deploy ahead of any demo, never live in front of a room: the cloud step takes a few minutes and occasionally argues.

Prompt 4 · deploy 
Walk me through deploying this agent to Azure with azd, one step at a time.

I'm not a developer, so for each step tell me exactly what to type and what I should
see when it works. If something needs an account, a login, or a one-time setup,
stop and tell me before I run it.

When it's deployed, show me how to add it to a Teams chat and test it with an @mention.
Path A · Teams + CLI Like Otto Deploy with azd, sideload into a Teams chat. The full-control route, what you saw live.
Path B · Copilot Studio No terminal Point-and-click builder. Same persona and knowledge, no command line. Publishes to Teams and Copilot.
Path C · M365 agents Inside Copilot Build an agent that lives in M365 Copilot for your team. Best when your knowledge already sits in SharePoint.

What to do this week

Clone the repo with Prompt 1. Rename it to your own agent.

Write the persona with Prompt 2, then drop in one folder of your team's knowledge with Prompt 3.

Get it running with Prompt 4. You don't have to ship it to the world. One agent, one team, working.
